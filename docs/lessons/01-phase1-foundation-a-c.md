# 教學文章 1 — Phase 1 地基（A→C）：資料遷移、漸進模組化、學習資料模型

> 對應 commit：`0c4befa`
> 範圍：A 資料地基（vocab 補 id）、B 前端模組化（store.js / progress.js）、C F0 題項精熟模型

---

## 一、這個階段在解什麼問題

使用者的痛點是「**看不出該題庫的學習進度**、練習沒效率」。
讀過程式碼後，根因不是 UI，而是**資料模型**：

- 進度只追蹤「測驗場次」（`recordQuizResult` 只累加 answered/correct/sessions），
  完全沒有「哪個字學會了沒」的**題項層級**資料。
- 出題是純隨機（`makeVocabQ` = `shuffle(pool)[0]`），已會的字和沒看過的字
  被抽中機率一樣。

所以 Phase 1 的第一件事不是做漂亮的進度條，而是**先把地基打好**：
讓每一個題項（單字／句子／文法題）都能被穩定識別、被追蹤掌握度。
這就是「F0 題項精熟模型」，以及它的前置——資料要有穩定 id。

> **關鍵觀念**：當使用者抱怨「看不到 X」，先問「我們的資料**記得** X 嗎？」。
> 很多 UI 問題其實是資料模型問題。先補資料模型，UI 才有東西可長。

## 二、A：為什麼要先做一次「資料遷移」

內建題庫 `data/levels/L1–L7.json` 的 vocab 長這樣：

```json
{ "es": "hola", "zh": "你好", "ipa": "ˈola", "cat": "問候與寒暄" }
```

**沒有 id**。listening / grammar 有 id，vocab 沒有。

要追蹤「這個字學會了沒」，得有一把**穩定的鍵**。兩個選項：

| 方案 | 成本 | 風險 |
|---|---|---|
| 用 `es` 字串當鍵 | 低（不動資料） | 改一個字的拼寫 → 學習歷史斷掉（被當新題項） |
| 補穩定 `id`（`vx_<LID>_NNN`） | 高（動 7 個 JSON + 工具 + 前端） | 一次性遷移要保證不弄壞既有資料 |

owner 選了**補 id**——用較高的一次性成本，換「日後改題庫內容、學習歷史不斷」的長期正確性。

### 一次性遷移腳本怎麼寫才安全

`tools/add_vocab_ids.py` 的設計重點，每一條都是可遷移到任何資料遷移的原則：

1. **冪等（idempotent）**：已有 id 的項目跳過、不重新編號。
   重跑一百次結果一樣 → 可以安心在不確定狀態下重跑。
2. **不重排**：嚴格依現有陣列順序編號，不動既有順序與內容。
3. **內容雜湊驗證**：遷移前後對 `(es,zh,ipa,cat)` 算 SHA-256，
   逐級比對**必須完全相同**。這證明「只加了 id，沒動到任何內容」。
4. **`--check` 乾跑模式**：先看會改什麼，再真的寫。
5. **寫回格式對齊**：`json.dumps(..., ensure_ascii=False, indent=2)`
   與既有寫檔工具完全一致 → diff 乾淨（每檔正好 +150 行 id、0 刪除），
   review 時一眼就知道沒有意外變更。

> **關鍵學習**：資料遷移的可信度來自「**可驗證的不變量**」，不是「我看起來沒問題」。
> 雜湊比對 + 冪等 + 乾跑，是三個低成本但高保證的習慣。

### 工具也要一起改

只改資料、不改產生資料的工具，下次工具一跑就破功。
所以 `tools/merge_seed.py` 的 vocab 分支也比照 `lx_/gx_` 邏輯補 `vx_` id。
**改資料時，連同寫資料的程式一起改**，否則就是埋一顆未爆彈。

## 三、B：漸進模組化——現代化但**不換框架**

owner 原本要「前後端轉現代架構」。但這個 app 有硬限制：
離線、無 build step、無 CDN、單檔可雙擊、markdown 資料夾儲存、單人本機。
評估後的結論是：**框架（React/Vite）換不到對等價值，卻會破壞這些硬限制**。
真正的痛點是「2020 行單檔可維護性」，不是技術棧。

所以做法是：用**原生 ES module**（`<script type="module">`）拆檔——
零 build、可離線、瀏覽器原生支援——而不是引入框架。

這一階段只抽兩個模組（Tier 0），刻意**不**全部重寫：

- `frontend/js/store.js`：極簡 reactive store（state + mutation + subscribe）。
- `frontend/js/progress.js`：進度狀態 + Markdown codec + 新的 F0 模型。

`index.html` 的 `<script>` 改成 `<script type="module">`，`import` 這兩個模組，
其餘 1400 行 App 邏輯**原地不動**。

### 兩個容易忽略的工程細節

1. **`type="module"` 是模組作用域，不是全域**。原本 `onclick="App.xxx()"`
   的 inline handler 找不到 `App`。解法：明確 `window.App = App;`。
   這類「換了載入方式，全域假設就破」的坑，重構時要主動想。
2. **模組怎麼被瀏覽器拿到**：後端用 FastAPI 把 `frontend/` mount 在 `/static`，
   所以 `import ... from '/static/js/progress.js'` 可被解析；
   `progress.js` 內 `import './store.js'` 相對解析到 `/static/js/store.js`。
   驗證時要確認 MIME 是 `text/javascript`（不是 `text/plain`，否則瀏覽器拒載 ES module）。

> **關鍵學習**：「現代化」不等於「換框架」。先界定**硬限制**，再選代價最低、
> 又能解真痛點（可維護性）的手段。重構**只服務於功能**、分層漸進，不獨立排期大重寫。

## 四、C：F0 題項精熟模型——地基的核心

`progress.js` 為每個題項記一筆：

```
{ seen, correct, wrong, streak, lastSeen, ease, due }
```

幾個設計決策值得學：

### 1. 穩定鍵 `itemKey` 的「內建 vs 自製」分流

```
builtin：`${typ}:${id}`            // 用 vx_/lx_/gx_ 穩定 id
custom ：`${bankKey}|${typ}|${es}` // 自製題庫無穩定 id，退回正規化 es
```

自製題庫每次載入給的是隨機 id（不穩定），所以**只有 builtin 才信任 id**，
否則退回 `es`。`itemKey()` 用一個明確的 `isBuiltin` 參數決定，不靠猜。

> **關鍵學習**：同一個概念（穩定鍵）在不同來源有不同保證時，
> 把「保證等級」變成函式的**顯式參數**，不要用啟發式猜測。

### 2. 持久化節奏：記憶體累計、整場一次寫檔

`answer()` 每題即時更新精熟（記憶體），但**不是每題寫檔**；
整場結束 `showResult → persistProgress` 才一次寫入 markdown / localStorage。
理由：避免練習中每題都做檔案 I/O；節奏也與原本「一場一寫」一致。

> **關鍵學習**：寫入頻率是一個設計決策。把「即時更新狀態」與
> 「持久化」分開思考，常常能用很小的代價避免效能與一致性問題。

### 3. 向下相容的 Markdown schema

進度存在 `progress/<bank>.md`。F0 在裡面**新增**一個 `# 題項精熟` 區塊。
解析器設計成**區段感知**：舊檔沒有這個區塊 → `items` 視為空物件，
既有 frontmatter 與「測驗歷史」表完全不動。

這讓「已經在用的人」升級後資料不會壞——這是個人工具長期演進的關鍵。

> **關鍵學習**：對**已部署的資料格式**做擴充時，預設要「新增、相容、可降級」，
> 而不是「改結構」。並用 round-trip 測試（格式化→解析→還原）守住它。

## 五、怎麼驗證（沒有瀏覽器也能有信心）

- `tools/test_data_invariants.mjs`（node --test，零依賴）：每級 vocab=150、
  id 皆 `vx_<LID>_NNN` 且唯一、id 為首鍵、内容非空。
- `tools/test_progress_model.mjs`：itemKey 規則、recordItem 計數、
  MD round-trip、**舊格式向下相容**。
- `tools/test_merge_seed_ids.py`：工具改完仍冪等、配 id 正確。
- 後端煙霧：`/`、`/static/js/*`、`/api/banks`（id 帶出、總數不變）。

這裡有個誠實的邊界：**自動測試（無瀏覽器）能覆蓋資料與模組行為，
但覆蓋不到「UI 在瀏覽器裡長怎樣」**。後面 F6 那顆「首載空白」的 bug
就是這個盲區的證明（見文章 3）。所以報告時要明白區分
「已自動驗證」與「需人工在瀏覽器確認」。

## 六、一頁帶走的重點

1. 使用者說「看不到」，先檢查「資料有沒有記」。UI 問題常是資料模型問題。
2. 資料遷移靠**冪等 + 不重排 + 內容雜湊 + 乾跑**建立可信度；改資料連工具一起改。
3. 現代化 ≠ 換框架。先定硬限制，用原生 ESM 漸進拆檔，重構只服務功能。
4. 穩定鍵的「信任等級」用顯式參數表達，別用猜的。
5. 對已部署格式只做「相容性擴充」，並用 round-trip 測試守住。
6. 誠實區分「自動已驗證」與「需人工驗證」，別把測試綠當成功能正確。
