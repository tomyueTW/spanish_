## 一、F5 在解什麼

`progress/<bank>.md` 早就存了每場測驗的歷史（`history[]`），
但 UI 從不呈現——使用者要看自己的趨勢，得自己去開 markdown 檔。
F5 把這些**已經存在、卻看不到的資料**變成一個「📈 學習進度」頁：

- 總計（場次／答題數／正確率）
- **連續學習天數**
- **近 16 場正確率走勢**（折線）
- 各題庫精熟度（重用 F1 的精熟條）

> **關鍵學習**：做新功能前先問「**這個資訊是不是其實已經存了、只是沒顯示**」。
> F5 幾乎沒產生新資料，只是把既有 `history` 可視化。**先榨乾已有資料的價值。**

## 二、硬限制下的視覺化：為什麼不引圖表庫

硬限制是「離線、無 CDN、無 build」。Chart.js / D3 這類庫違反這些限制
（要嘛 CDN、要嘛 npm/bundle）。而需求其實很小：一條正確率折線。

解法：**手寫內聯 SVG sparkline**，零依賴。核心就是把資料點映射到座標：

```js
const W = 240, H = 56, pad = 4, n = vals.length;
const x = i => pad + (W - 2*pad) * (i / (n - 1));      // 索引 → x
const y = v => pad + (H - 2*pad) * (1 - v / 100);      // 0–100% → y（上下翻轉）
// → <polyline points="..."> + 末點一個 <circle>
```

幾個值得記的細節：

- `viewBox` + `preserveAspectRatio="none"`：用 viewBox 座標系畫圖，
  讓 SVG 隨容器寬度自由縮放，不用算實際像素。
- `vector-effect="non-scaling-stroke"`：縮放時線寬不被一起拉伸，維持銳利。
- 資料點 < 2 時不畫線，改顯示「資料不足」——**邊界狀態要主動處理**，
  否則 `i/(n-1)` 在 `n=1` 會除以 0。
- `role="img"` + `aria-label`：純圖形也給可及性描述。

> **關鍵學習**：
> 1. 在引入相依前先問「**需求的最小核心**有多大」。一條折線 ≈ 一個座標映射
>    + 一個 `<polyline>`，不值得為它違反硬限制扛一個圖表庫。
> 2. 視覺化的本質就是「資料 → 座標」的函式。把它寫成兩個純函式（`x()`、`y()`），
>    程式清楚、邊界（n<2、除零）也好處理。

## 三、連續學習天數：看似簡單，全是邊界

「連續幾天有練習」要從一堆 session 的時間戳算出。難的不是迴圈，是邊界：

1. **去重到「日」**：同一天練 5 場算 1 天 → 先把時間戳 `slice(0,10)` 取日期、去重。
2. **「今天還沒練」不該歸零**：如果最後一次練習是昨天、今天還沒練，
   連續天數不應變 0（不然使用者一早打開看到 streak 歸零會很挫折）。
   做法：若最近練習日 < 今天，**從昨天起算**。
3. **用 UTC 毫秒比對日界**：把 `YYYY-MM-DD` 轉成 `Date.UTC(y,m-1,d)` 再
   逐日 `-86400000` 往回找，避免本地時區與日光節約把「同一天」算成兩天。

> **關鍵學習**：「連續天數」這種需求，**90% 的工作量在邊界與人因**：
> 跨日界、時區、今天還沒做要不要歸零。寫之前先把這些情境列出來，
> 比寫完再 debug 省時。人因（歸零的挫折感）也是規格的一部分。

## 四、誠實資料：不畫我們沒存的東西

規劃書原本寫「精熟曲線（隨時間精熟項數）」。但我們**從未儲存精熟度的時間序列**——
只有「當前」精熟分佈。要畫「精熟隨時間成長」就得**假造**或回推資料。

選擇：**不畫**。F5 只呈現真實存在的東西——正確率走勢（來自真實 history）、
連續天數（來自真實日期）、當前精熟分佈（當前真值）。

> **關鍵學習**：視覺化最大的誘惑是「補一條看起來很厲害但資料不存在的曲線」。
> **寧可少一張圖，也不要畫一張需要假造資料才成立的圖。**
> 如果未來真要「精熟成長曲線」，正確做法是先在資料層**開始記錄每日精熟快照**，
> 而不是在 UI 層回推。視覺化不能比資料誠實。

## 五、一個小而真實的工程坑：id 命名衝突

新頁面原本想叫 `id="progress"`。但 `index.html` 早就有
`<div class="progress" id="progress">`（測驗中的進度文字）。
若沿用會造成 `document.getElementById('progress')` 指向錯元素、
`show()` 的頁面清單也會打架。改用 `id="progress-page"`，
並在驗證時用 grep 確認 `id="progress"` 仍只出現該用的地方。

> **關鍵學習**：在大型單一 HTML 檔加新頁面，**先 grep 你要用的 id 是否已存在**。
> 全域 id 命名空間在單檔 app 是共享資源，衝突的症狀（抓到別的元素）
> 往往詭異又難查。命名加上語意後綴（`-page`）可降低碰撞。

## 六、一頁帶走的重點

1. 做新功能前先問「資料是不是已經存了、只是沒顯示」，先榨乾既有資料。
2. 引相依前先量「需求最小核心」；一條折線不值得違反硬限制。
3. 視覺化＝資料→座標的純函式；邊界（n<2、除零）要主動處理。
4. 「連續天數」90% 工作在邊界與人因（跨日界、時區、今天未做不歸零）。
5. 視覺化不能比資料誠實——寧可少一張圖，不畫需假造資料的圖。
6. 單檔 app 加頁面先 grep id 是否撞名；id 加語意後綴防碰撞。
