# 教學文章 2 — Phase 1 D–H：把「測驗器」變成「學習系統」

> 對應 commit：`4d13c74`
> 範圍：D1 設定基建、D F1 精熟總覽、E F2 弱項加權、F F3 錯題複習、G F7 鍵盤、H 後端 reload

---

## 一、為什麼這些功能要綁在一起做

A→C 打好地基（每個題項有穩定鍵、有掌握度資料），但**使用者還看不到任何改變**。
D–H 是把地基「變現」成使用者價值的一組功能，它們有共同依賴（F0 精熟資料），
一起做最省：

- **D（F1）** 讓進度**看得見**（精熟條）。
- **E（F2）** 讓練習**有方向**（弱項優先）。
- **F（F3）** 讓錯題**不再答完即焚**（錯題本／複習）。
- **G（F7）** 讓高頻操作**更順手**（鍵盤作答）。
- **H** 消除 owner 自己加題時「要重啟後端」的摩擦。

> **關鍵學習**：地基功能（看不見的資料層）和變現功能（看得見的價值）要分階段，
> 但**變現功能彼此若共用同一份資料，應集中在同一階段**，避免反覆改同一處。

## 二、D1：先做「設定」這個不起眼的基建

F1 要顯示精熟，得先定義「連續答對幾次算精熟」。owner 拍板：**3 次，可在設定調整**。
F2 要能開關「弱項優先」。所以先做一個 `settings` 物件：

```js
let settings = { masteryStreak: 3, weightedSelection: true };
```

透過既有的 `saveState/loadState` 持久化到 localStorage 的 `settings` 鍵，
讀取時做**範圍驗證**（`masteryStreak` 限 2–5），UI 用既有 `.settings-row`/`.num-select`
樣式，不新增視覺元件。

> **關鍵學習**：使用者要的是「精熟條」，但要先做「設定的儲存/讀取/驗證」這種
> 沒人會稱讚的基建。**先建承載決策的基建，功能才有地方掛。**
> 並且：可調參數一律「預設值 + 範圍驗證 + 持久化」三件套。

## 三、E（F2）最值得學：把「挑題」與「建題」拆開

原本 `makeVocabQ` 把「選哪個字」和「組成一道題」混在一起：

```js
const shuf = shuffle(pool); const correct = shuf[0]; // 選 + 用 糾纏
```

要做「弱項優先」必須改「選哪個字」，但又不想動「怎麼組題」。於是重構成：

- `pickTarget(pool, typ)`：**只**負責挑焦點題項（加權或均勻隨機）。
- `buildVocabQ(correct, pool)` / `buildListenSentenceQ` / …：**只**負責由指定
  題項組成一道題。
- `makeVocabQ` 變成薄薄一層：`buildVocabQ(pickTarget(pool,'vocab'), pool)`。

這個「**選擇/建構分離**」不只服務 F2，也讓 F3 複習能「對指定錯題強制出題」
（直接呼叫 `buildXFrom`）。一個好的接縫，常常一次解鎖多個功能。

加權公式（`progress.js` `itemWeight`）：

```
未學 6  >  答錯過 4  >  學習中 2  >>  已精熟 0.5（低但非 0，仍會偶爾複習）
```

> **關鍵學習**：當你要改「行為的一部分」卻怕動到其他部分，先找出
> **正交的接縫**（這裡是「挑 vs 組」），沿縫切開。好接縫會順帶解鎖未來需求。
> 另外：弱項加權給已精熟項「低但非 0」的權重——**不要把已學的歸零**，
> 防遺忘也是學習效率的一部分。

## 四、F（F3）：錯題的「定義」就是設計

「錯題」不是「曾經答錯」這麼簡單。定義成：

```
wrong > 0  且  streak < 精熟門檻
```

也就是「答錯過、而且**還沒重新學會**」。重新精熟後自動移出錯題清單——
這讓「複習錯題」會隨著你變強而**自然清空**，而不是無止境累積。

入口有兩個（都沿用既有元件，不新增視覺語言）：
主選單「🔁 複習錯題」mode-btn、結果頁「立即複習這些」。

> **關鍵學習**：一個功能的品質，往往藏在它對核心名詞的**定義**裡。
> 「什麼算錯題」決定了複習體驗會不會讓人有成就感。定義先於 UI。

## 五、G（F7）：可及性是高頻體感

每天練的人用滑鼠點 1–4 很煩。加 1–4 鍵選答、Enter/Space 下一題。
實作沿用 browse 模式既有的 `bindXKeys/unbindXKeys` 模式，
在 `show()` 進出 quiz 時綁定/解綁，並忽略 INPUT/TEXTAREA 來源。

> **關鍵學習**：低成本、高頻率的體感改善（鍵盤）優先序不該被低估。
> 沿用程式裡**已存在的模式**（這裡是按鍵綁定的生命週期管理），
> 一致性本身就是品質，也降低出錯面。

## 六、H：消除作者自己的摩擦

owner 也是內容作者。改 `data/levels/*.json` 後要重啟後端才生效，是天天踩的摩擦。
`backend/main.py` 把載入抽成 `load_levels()`，加 `POST /api/reload` 重載並回計數。
低風險、不動 `/api/banks` 熱路徑。

> **關鍵學習**：把「**維護者自己的摩擦**」也當成產品問題。
> 工具鏈的小摩擦累積起來，會悄悄拖慢整個專案的迭代速度。

## 七、誠實的部分：自審抓到的 bug

F3 複習時，`bankItemIndex` 與出題 pool 來自**兩次不同的 `getActiveBank()` 呼叫**，
而 `getActiveBank()` 每次 `.map(w => ({...w}))` 產生**新物件**。
`buildVocabQ` 用 `o !== correct` 以**物件 identity** 排除正解當干擾項——
但兩邊物件不同實例，identity 比對失效 → 可能出現**兩個相同選項**，
答案索引指向其一，使用者點另一個會被判錯。

修法：讓 index 與 pool 共用**同一個** bank 物件（`bankItemIndex(bank)` 吃參數）。

> **關鍵學習**：
> 1. 用「物件 identity」(`!==`) 當邏輯依據時，要確定兩邊真的是同一個實例；
>    一旦中間有 `.map`/`{...spread}`/序列化，identity 就斷了。
> 2. 沒有瀏覽器自動化時，**逐函式自我審查資料流**（誰產生物件、誰比對、
>    生命週期是否一致）能抓到測試覆蓋不到的真 bug。把這當交付前的固定動作。

## 八、一頁帶走的重點

1. 地基要「變現」成使用者價值；共用同一份資料的功能集中一個階段做。
2. 可調參數＝預設值＋範圍驗證＋持久化；先建承載決策的基建。
3. 找正交接縫（挑題 vs 組題）再切，好接縫一次解鎖多功能。
4. 功能品質藏在核心名詞的定義（「什麼算錯題」）。
5. 已精熟項權重「低但非 0」——防遺忘也是效率。
6. 維護者自己的摩擦也是產品問題。
7. `!==` identity 判斷要確認同實例；交付前逐函式自審資料流。
