# 教學文章 4 — Phase 2 F4：輕量間隔複習（Leitner 盒）

> 對應 commit：`436e024`
> 範圍：間隔複習理論、Leitner vs SM-2 取捨、box/due 資料模型、日期處理陷阱、向下相容的格式擴充

---

## 一、為什麼要做間隔複習（SRS）

F2 讓你「多練沒學會的」，但**學會之後呢**？人會遺忘。
間隔複習（Spaced Repetition）的核心事實：

> 在「快要忘記」的時間點再複習一次，記憶保留最久；
> 學得越熟，下次該複習的間隔就拉得越長。

沒有 SRS，使用者只能「憑感覺」決定今天練什麼，常常一直練已經很熟的、
或讓真正快忘的悄悄流失。F4 給每個題項一個「**下次該複習日**」，
主選單直接告訴你「今天有 N 題該複習了」。

---

## 二、關鍵取捨：Leitner vs 完整 SM-2

這是規劃書明列、需 owner 拍板的決策（§5 Q3）。兩個選項：

| | Leitner 盒 | 完整 SM-2（SuperMemo-2） |
|---|---|---|
| 模型 | 盒 1..5，答對升盒、答錯回盒 1；盒對應固定間隔 | 每題維護浮點 ease factor + interval，依作答品質動態調整 |
| 可解釋性 | 高（「升盒/降盒」一句話講完） | 低（ease 怎麼算、為什麼這個間隔，難向自己解釋） |
| 調參 | 無（間隔表是固定常數） | 需調 ease 公式參數 |
| 對單人學習者 | 足夠 | 過度工程 |
| 測試成本 | 低 | 高（浮點、邊界多） |

owner 選 **Leitner**。間隔預設 `1 / 3 / 7 / 16 / 35` 天（可調）。

> **關鍵學習**：演算法選型不是「哪個更聰明」，而是「哪個的**複雜度匹配情境**」。
> 單人、本機、要能自己解釋的工具，Leitner 的「可解釋、零調參」勝過 SM-2 的精準。
> **過度工程的精準，是一種看不見的負債。**

## 三、資料模型：複用既有預留欄位

F0（文章 1）當初就把 `due`、`ease` 兩個 SRS 欄位**先放著、不啟用**。
F4 啟用 `due`、新增 `box`，`ease` 維持保留（給未來 SM-2 留路，但現在不碰）。

```
LEITNER_INTERVALS = [_, 1, 3, 7, 16, 35]   // 以 box 當索引（box0 不用）
答對：box = min(box+1, 5)
答錯：box = 1
due  = 作答日 + LEITNER_INTERVALS[新 box] 天
```

> **關鍵學習**：地基階段為「**可預見但還不做**」的功能預留欄位（不啟用），
> 是低成本的前瞻。等真的要做時，schema 已經相容，遷移成本接近零。

## 四、最容易忽略、也最值得學的坑：非法日期

`due` 由「作答時間 + 間隔天數」算出。最直覺的寫法：

```js
const d = new Date(nowIso); d.setDate(d.getDate() + interval);
return d.toISOString().slice(0, 10);
```

問題：既有的 `progress.js` 單元測試為了簡潔，很多地方傳的是
`recordItem(..., 't')`——一個**非法日期字串**。`new Date('t')` 是 Invalid Date，
`.toISOString()` 會**丟 RangeError**。如果不處理，F4 一上線，
**既有 30+ 個測試會全部爆掉**——而且爆的不是 F4 的錯，是「我改了共用函式
卻沒考慮既有呼叫者的輸入假設」。

修法是顯式守衛：

```js
const d = new Date(nowIso);
if (isNaN(d.getTime())) return '';   // 非法 → 不算 due，不破壞既有測試
```

> **關鍵學習**：
> 1. 修改**被廣泛呼叫的共用函式**時，先盤點現有呼叫者餵什麼輸入
>    （包含測試裡的「偷懶輸入」）。你的新邏輯要對既有輸入**優雅降級**，
>    而不是要求全世界配合你。
> 2. 日期運算永遠先想「**非法／邊界輸入**」：Invalid Date、時區、
>    跨月跨年。`new Date(x)` 不丟例外但會產生 Invalid Date，
>    真正丟例外的是後面的 `.toISOString()`——錯誤會在離源頭很遠的地方爆。

「到期」判定也有日期細節：用 `YYYY-MM-DD` 字串比較
（`dueKeySet` 裡 `it.due <= today`）。為什麼可行？因為 ISO 日期字串的
**字典序＝時間序**。這是個常用技巧：能用字串比就不要轉 Date 物件，少一層出錯面。

## 五、又一次向下相容的格式擴充

`# 題項精熟` 表原本 6 欄，F4 要多存 `盒`、`到期` → 變 8 欄。
舊的 `progress/<bank>.md` 還是 6 欄。解析器必須**同時吃 6 欄與 8 欄**：

```
8 欄 → 完整解析 box/due
6 欄 → 舊檔：box 預設 1、due 空字串（向下相容）
其它欄數 → 跳過
```

這是文章 1「對已部署格式只做相容性擴充」原則的第二次實踐。
每次擴充欄位都加對應的 round-trip 測試 + 舊格式解析測試，
讓「相容性」變成有測試保證的不變量，而不是口頭承諾。

> **關鍵學習**：持久化格式會隨功能長大。把「**舊版讀得動新版產生前的檔**」
> 當成硬性需求，並用測試釘死。個人長期工具，使用者就是你自己——
> 升級弄壞自己的歷史資料是最不可原諒的 bug。

## 六、串起來的使用者價值

- `dueKeySet(bankKey, now)`：已作答且 `due <= 今天` 的題項。
- F6 的 hub `#due-banner`：「📅 今天有 N 題該複習了」＋一鍵開練。
- 新增 `'due'` 出題模式，與 `'review'` 共用 `buildQuestionsFromKeys`
  （文章 2 那個「選擇/建構分離」接縫又一次被複用——好接縫的價值在此）。

## 七、一頁帶走的重點

1. SRS 的價值是「在快忘時複習」；沒有它，使用者只能憑感覺。
2. 演算法選型看「複雜度是否匹配情境」，不是看誰更聰明；過度精準是負債。
3. 地基階段為可預見功能**預留不啟用的欄位**，未來遷移近乎零成本。
4. 改共用函式先盤點既有呼叫者輸入；對偷懶/非法輸入要優雅降級。
5. 日期：先想 Invalid Date／時區／跨月；能用 ISO 字串字典序比較就別轉 Date。
6. 持久化格式擴充永遠「相容、可降級」，並用 round-trip + 舊格式測試釘死。
